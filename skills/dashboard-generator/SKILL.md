# 仪表板生成器

结构化工作流，用于设计数据仪表板。自动检测用户意图，从合适的步骤开始。

## 自适应工作流

```
用户输入
    │
    ├─ 已有明确布局规格（组件、位置、尺寸）? ──→ 步骤4 (布局渲染)
    │
    ├─ 已有明确指标 + 维度? ───────────────────→ 步骤3（布局设计）
    │
    ├─ 已有明确目标但无指标? ──────────────────→ 步骤2（指标体系）
    │
    └─ 模糊或全新需求? ───────────────────────→ 步骤1（需求梳理）
```

### 意图检测规则

| 用户提供的信息 | 起始步骤 | 原因 |
|---|---|---|
| 仅主题（"销售仪表板"） | 步骤1 | 需要明确受众、目标、问题 |
| 已有目标 + 受众 | 步骤2 | 可以推导指标 |
| 已有指标 + 维度 | 步骤3 | 可以设计布局 |
| 已有明确布局（组件、网格、位置） | 布局渲染 | 跳过设计，直接输出 |
| 改进现有仪表板 | 评估缺口 | 识别哪个步骤有不足 |

开始前声明:

> "根据你的输入，我从步骤[N] — [步骤名称]开始。[简要原因]。"

## 执行钩子

请在步骤执行后， 输出相关信息
> "当前步骤[N] - 思考：[思考]"

## 步骤1: 需求梳理

明确谁看仪表板、为什么需要、要回答什么问题。

1. 识别受众（角色、数据素养、决策频率）
2. 定义仪表板用途（监控 / 分析 / 决策支持）
3. 列出关键业务问题（3-7个仪表板必须回答的问题）
4. 确定约束条件（刷新频率、数据源、访问范围）

使用 [references/analysis-template.md](references/analysis-template.md) 中的需求模板输出。完成后进入步骤2。

## 步骤2: 指标体系（GSM模型）

使用 目标 → 信号 → 指标 框架从目标推导指标。

1. 定义业务目标（来自步骤1的问题或用户输入）
2. 识别信号 — 表明目标进展的可观察行为
3. 推导指标 — 每个信号的可量化度量
4. 分配维度 — 切分指标的分析轴

详见 [references/metrics-framework.md](references/metrics-framework.md) 获取GSM模式和行业示例。

使用 [references/metrics-template.md](references/metircs-template.md) 中的指标模板输出。完成后进入步骤3。

## 步骤3: 布局设计

设计仪表板网格、选择可视化类型、定义交互。

1. 指标优先级排序（P0 = 顶层KPI, P1 = 支撑指标, P2 = 下钻明细）
2. 为每个指标选择图表类型
3. **步骤4: 布局渲染（信息层级、视觉流向）**

## 步骤4： 布局渲染

当用户提供明确的布局规格（组件列表、位置、尺寸、图表类型）时，跳过设计流程直接输出:

1. 将用户的布局规格解析为组件映射表
2. 生成匹配规格的ASCII线框图
3. 根据组件类型添加交互定义
4. 匹配布局（信息层级、视觉流向）
- ([检测布局类型](references/layout-detect.md))
根据返回的布局类型，通过布局限制，进行布局生成。
- 如果是流式布局，则参考[流式布局限制](references/layout-stream-constraints.md)
- 如果是tab布局，则参考[tab 布局限制](references/layout-tab-constraints.md)

尊重用户的布局决策，无需质疑或重新设计。仅在存在明显可用性问题时建议改进（如多维度仪表板缺少筛选器）。

## 步骤5：布局验证

最终的输出需要通过验证。验证循环 2 次。如果失败，则直接交付

- [流式布局验证](references/layout-stream-validate.md)
- [tab 布局验证](references/layout-tab-validate.md)
